들어가며

웹사이트를 개발하다 보면 클라이언트가 보낸 데이터를 객체에 한 번에 담아주는 Mass Assignment 기능을
자주 사용하게 된다.

이 기능은 개발 생산성을 크게 높여주지만 잘못 사용하면 보안 취약점으로 이어질 수 있어 조심해야 한다.

오늘은 Mass Assignment라는 기능이 무엇인지, 그리고 왜 문제가 되는지 정리해보았다!

사실은 CTF나 워게임을 풀 때 Mass Assignment 취약점을 자주 만났었는데 시간이 지나고 계속 까먹고 또 까먹고를
반복해서 머리에 확실하게 저장하려고 블로그 글을 쓴다.

Mass Assignment란?

Mass Assignment는 클라이언트가 전송한 요청 데이터를 객체의 여러 필드에 한 번에 할당하는 기능을 의미한다.

예를 들어 회원가입을 위해 사용자가 다음과 같은 데이터를 보냈다고 가정해보자.


{
   "username": "yunseul", 
   "email": "y7nse4l@gmail.com"
}

서버는 이 데이터를 User 객체에 매핑하여 저장한다.

이때 저 데이터들을 user.setUsername(request.username), user.setEmail(request.email) 과 같이 각 setter 함수들을 호출하여 저장할 수도 있지만,
대부분의 웹 프레임워크는 편의성을 위해 요청 데이터를 자동으로 객체에 한 번에 할당해주는 기능을 제공한다.

예를 들어, Spring에서는 다음과 같이 사용할 수 있다.


public class User { 
	private String username; 
	private String email; 
	private boolean isAdmin; 
}

@PostMapping("/users")
public User createUser(@RequestBody User user) {
	return userRepository.save(user);
}

여기서 @RequestBody User user 코드가 실행되는 순간, Spring은 HTTP 요청 데이터를 User 객체에 자동으로 바인딩한다.

즉, 개발자가 각 필드를 직접 꺼내지 않아도 프레임워크가 알아서 객체를 생성하고 값을 채워준다.
이 기능 덕분에 코드가 간결해지고, 객체에 필드가 늘어나더라도 핸들러 코드를 수정할 일이 줄어든다.

무엇이 문제인가 - Mass Assignment Vulnerability

여기까지만 보면 그저 편리한 기능일 뿐이다. 하지만 사용자가 보낸 데이터를 검증 없이 바로 객체에 바인딩할 때 문제가 발생한다.
다시 User 객체를 살펴보자.


public class User { 
	private String username; 
	private String email; 
	private boolean isAdmin; 
}

개발자가 프론트엔드에서 username과 email만 입력할 수 있도록 구현했다고 가정하자.
하지만 @RequestBody User user는 요청에 들어온 모든 필드를 객체에 바인딩한다.
서버 입장에서는 화면에 어떤 입력 창이 있는지 알지 못하기 때문이다.

따라서 공격자가 직접 HTTP 요청을 조작하여 다음과 같은 데이터를 보낼 수 있다.


{
   "username": "yunseul", 
   "email": "y7nse4l@gmail.com",
   "isAdmin": true
}

그러면 Spring은 내부적으로 다음과 같은 동작을 수행한다.


user.setUsername("yunseul"); 
user.setEmail("y7nse4l@gmail.com"); 
user.setAdmin(true);

이렇게 되면 일반 사용자가 관리자 권한을 획득할 수 있게 된다.

이처럼 사용자가 수정해서는 안 되는 필드까지 자동으로 객체에 매핑되는 취약점을 Mass Assignment Vulnerability라고 한다.

Mass Assignment Vulnerability 예시: 2012년 Github 사건

가장 유명한 Mass Assignment 취약점 사례는 2012년에 발생한 Github 사건이다.

당시 Github에는 사용자의 공개키(SSH Key)를 등록하는 기능이 있었는데, 공격자는 API 요청에 원래 의도되지 않은
파라미터를 추가하여 다른 사용자의 저장소와 자신의 공개키를 연결하는 데 성공하였다.
결과적으로 공격자는 해당 저장소에 대한 Push 권한을 획득할 수 있었다.

이 사건의 발단은 한 개발자 Egor Homakov였다.
그는 Github가 사용하던 Ruby on Rails의 Mass Assignment 기능의 위험성을 여러 차례 알렸지만 심각하게 받아들여지지 않자, 결국 직접 취약점을 시연했다고 한다.

대응 방법

핵심 원칙은 하나이다.
클라이언트가 보낸 입력을 도메인 객체에 곧바로 바인딩하지 말 것! OWASP가 권장하는 방법을 정리해보았다.

1. DTO로 입력 객체 분리하기

입력 전용 객체(DTO)를 따로 만드는 것이다.
사용자가 수정해도 되는 필드만 DTO에 담고, DTO를 통해 받은 값을 필요한 필드에 직접 매핑한다.
Spring MVC를 예시로 들면 다음과 같다.


 @Entity
  public class User {
      @Id @GeneratedValue
      private Long id;
      private String username;
      private String email;
      private boolean isAdmin;
  }

  // 입력으로 허용할 필드만 담은 DTO
  public class UserUpdateRequest {
      private String username;
      private String email;

      public String getUsername() { return username; }
      public String getEmail() { return email; }
  }
  
  @PutMapping("/users/{id}")
  public User updateUser(@PathVariable Long id,
                         @RequestBody UserUpdateRequest req) {
      // 1. 기존 사용자 조회
      User user = userRepository.findById(id).orElseThrow();

      // 2. DTO의 값을 도메인 객체로 직접 매핑 (isAdmin은 매핑하지 않음)
      user.setUsername(req.getUsername());
      user.setEmail(req.getEmail());

      // 3. 저장
      return userRepository.save(user);
  }

2. Allowlist 사용

바인딩 가능한 필드를 명시적으로 허용 목록으로 지정하여 사용하는 방법이다.
이렇게 하면 허용된 필드 외의 값을 무시할 수 있다. Spring MVC를 예시로 들면 다음과 같다.


// 폼/쿼리 파라미터(@ModelAttribute)를 받는 경우
  @InitBinder
  public void initBinder(WebDataBinder binder) {
      binder.setAllowedFields("username", "email"); // 이 필드들만 허용
  }

@InitBinder는 @ModelAttribute 바인딩에만 적용된다.
따라서 이 글의 createUser처럼 @RequestBody(JSON)로 받을 때에는 DTO 방식이나 아래 blacklist 방식을 사용한다.

3. Blacklist 사용

특정 필드를 차단하는 방식도 가능하지만, 새롭게 민감한 필드가 추가됐을 때 누락될 위험이 있으므로 Blacklist 방식보다는 1번의 DTO 방식을 권장한다.
Spring MVC를 예시로 들면 다음과 같다.


public class User {
      private String username;
      private String email;

      // 응답엔 나가지만 요청으로는 들어올 수 없음
      @JsonProperty(access = JsonProperty.Access.READ_ONLY)
      private boolean isAdmin;
  }
